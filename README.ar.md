مكتبة تكتشف محارف يونيكود الخطرة في النصوص التي يكتبها المستخدمون وتطهرها منها، دون أن تكسر نصا حقيقيا بالعربية أو العبرية أو غيرهما من لغات RTL، ودون أن تؤثر على رموز الإيموجي.

```bash
npm install unicode-shield
```

## لماذا هذه المكتبة

توجد محارف في يونيكود تظهر على الشاشة وكأنها غير موجودة، أو تجعل النص يظهر بشكل مختلف عن حقيقته الفعلية. اسم مستخدم، أو رسالة في محادثة، أو اسم ملف كتبه شخص خبيث، قد يخفي محارف:

- **تعكس اتجاه عرض النص** باستخدام محارف bidi embedding/override، وهي الآلية نفسها التي قام عليها الكشف الأمني المعروف باسم "Trojan Source" عام 2021 ([CVE-2021-42574](https://nvd.nist.gov/vuln/detail/CVE-2021-42574)). وهذه هي فئة الثغرة نفسها التي أصلحت في [`bluesky-social/social-app#11066`](https://github.com/bluesky-social/social-app/pull/11066): معرف مستخدم مثل `admin` يمكن أن تضاف إليه محارف مخفية تجعله عند نسخه ولصقه يظهر كنص مختلف تماما.
- **تحشو أو تكرر نصوص الهوية** باستخدام محارف عديمة العرض، بحيث يقارن اسما مستخدمين متطابقين بصريا على أنهما مختلفان، أو تتجاوز كلمة محظورة عامل تصفية.
- **تهرب تعليمات مخفية** باستخدام كتلة Unicode Tags المهجورة (U+E0000-U+E007F) أو Variation Selectors Supplement (U+E0100-U+E01EF)، وكلتاهما استخدمتا منذ عام 2024 كوسيلة لحقن أوامر خفية: نص مشفر في هذه النقاط يظهر فارغا في أي خط عادي، لكن بعض نماذج الذكاء الاصطناعي ما زالت تقرأه وتتصرف بناء عليه.

## لماذا لا نستخدم أداة موجودة بالفعل

توجد ماسحات للكود المصدري لهذه الفئة من المحارف، وهي مبنية جيدا، وعلى رأسها [`anti-trojan-source`](https://github.com/lirantal/anti-trojan-source). لكنها تحل مشكلة مختلفة: اكتشاف هذه المحارف *داخل الكود نفسه*. أما `unicode-shield` فمخصصة للجانب الآخر: اكتشافها *في البيانات التي يكتبها المستخدمون*، لحظة تخزينها أو عرضها.

قبل بناء هذه المكتبة، بحثت عن مكتبة تعمل وقت التشغيل تقوم بهذا، عبر npm وPyPI وcrates.io وpkg.go.dev، وكذلك أدوات الأمان مفتوحة المصدر التي تنشرها كبرى شركات الذكاء الاصطناعي. ما وجدته كان إما ضيق النطاق (يقتصر على إزالة المحارف عديمة العرض، أو على اكتشاف التشابه البصري فقط)، أو ماسح كود مصدري/CI يطبق على الطبقة الخاطئة، أو مشاريع حديثة جدا تكتشف المحارف الصحيحة لكنها تحذف المحارف الشرعية أيضا، فتكسر بالضبط نص RTL والإيموجي الذي تحرص هذه المكتبة على عدم المساس به. لم أجد أداة تجمع بين اكتشاف bidi، واكتشاف كتلة Unicode Tags، والحفاظ المتعمد على علامات الاتجاه الشرعية وروابط النص. هذا الجمع، بما في ذلك الدقة في الجانب الآمن، هو ما بنيت هذه المكتبة من أجله.

## الجانب الآخر من المشكلة

من السهل أن تخطئ أداة كهذه في الاتجاه المعاكس: حذف *كل شيء غير مألوف* وإفساد نص حقيقي دون أن تلاحظ. للعديد من المحارف المذكورة أعلاه استخدامات شرعية تماما:

- `‎` (U+200E، LEFT-TO-RIGHT MARK) و`‏` (U+200F، RIGHT-TO-LEFT MARK) هما تلميحا اتجاه بمحرف واحد يعتمد عليهما فعلا النص العربي والعبري وغيرهما من لغات RTL الصحيحة.
- `‍` (U+200D، ZERO WIDTH JOINER) هو الطريقة التي يبنى بها إيموجي مركب مثل 👨‍👩‍👧‍👦، وهو ضروري لتشكيل النص الصحيح في عدة لغات.
- `‌` (U+200C، ZERO WIDTH NON-JOINER) ضروري لتكوين الكلمة الصحيح في الفارسية وعدة لغات هندية.

تبلغ `unicode-shield` عن هذه المحارف أيضا، لأن الرؤية الكاملة لمحتوى النص مفيدة، لكنها لا تحذفها افتراضيا أبدا. تحذف `sanitize()` فقط الفئات التي ليس لها استخدام شرعي في نص قصير كتبه مستخدم.

## طريقة الاستخدام

```ts
import {scan, sanitize, isSafe} from 'unicode-shield'

isSafe('admin')                    // true
isSafe('admin\u{202E}nimda')       // false، يحتوي على bidi override

sanitize('admin\u{202E}nimda')     // 'adminnimda'

scan('price: 100\u{200E}\u{200F} ريال\u{061C}')
// {
//   safe: true,   // علامات bidi معلوماتية فقط، لا خطر حقيقي
//   threats: [
//     { category: 'bidi-mark', severity: 'informational', codePoint: 0x200e, ... },
//     { category: 'bidi-mark', severity: 'informational', codePoint: 0x200f, ... },
//     { category: 'bidi-mark', severity: 'informational', codePoint: 0x61c, ... },
//   ]
// }
```

### `scan(input: string): ScanResult`

تعيد كل تهديد موجود في `input`، الخطر والمعلوماتي معا. تكون `safe` بقيمة `true` فقط إذا لم يوجد أي تهديد خطر.

### `sanitize(input: string, options?: SanitizeOptions): string`

تعيد `input` بعد إزالة المحارف الخطرة منه. تحذف افتراضيا فئات `bidi-embedding` و`bidi-isolate` و`invisible` و`tag` و`variation-selector`. أما الفئات المعلوماتية (`bidi-mark`, `joiner`) فلا تمس إلا إذا طلب ذلك صراحة:

```ts
sanitize(input, {categories: ['bidi-mark']})   // تحذف أيضا LRM/RLM/ALM
sanitize(input, {replacement: '�'})       // تستبدل بدلا من أن تحذف
```

### `isSafe(input: string): boolean`

اختصار لـ `scan(input).safe`.

## فئات التهديد

| الفئة | الخطورة | أمثلة | تحذف افتراضيا؟ |
| ----------------- | --------------- | -------------------------------------- | -------------------- |
| `bidi-embedding`  | خطرة       | LRE, RLE, LRO, RLO, PDF                | نعم |
| `bidi-isolate`    | خطرة       | LRI, RLI, FSI, PDI                     | نعم |
| `invisible`       | خطرة       | zero-width space, word joiner, stray BOM | نعم |
| `tag`             | خطرة       | U+E0000-U+E007F (كتلة Tags المهجورة) | نعم |
| `variation-selector` | خطرة    | U+E0100-U+E01EF (Variation Selectors Supplement) | نعم |
| `bidi-mark`       | معلوماتية   | LRM, RLM, ALM                          | لا |
| `joiner`          | معلوماتية   | ZWJ, ZWNJ                              | لا |

ملاحظة حول `variation-selector`: كتلة الـSupplement فقط هي المشمولة. أما كتلة Variation Selectors الأساسية (U+FE00-U+FE0F، VS15/VS16) فلا تعد خطرة أبدا، لأنها الطريقة التي يختار بها النص العادي عرضا نصيا أو عرض إيموجي لآلاف المحارف الشائعة، وهي شائعة جدا في نصوص المستخدمين الحقيقية.

## إضافة ESLint

لا تفيد `scan`/`sanitize` إلا إذا استدعاهما أحد فعلا. والطريقة الأخرى التي تصل بها هذه الفئة من الأخطاء إلى الإنتاج هي مسار بيانات لا يمر بأي منهما إطلاقا: تكتشف `unicode-shield/eslint-plugin` هذه الحالة وقت المراجعة بدلا من الإنتاج، من خلال التحقق من وصول نص شبيه بالهوية (اسم مستخدم، أو handle، أو اسم عرض، أو bio) إلى JSX دون تطهير. وهذا فحص لبنية الكود من النوع الموصوف أعلاه في "لماذا لا نستخدم أداة موجودة بالفعل" بوصفه ماسح كود مصدري/CI، وليس الفحص وقت التشغيل الذي تقوم به `scan()`، وعلى عكس ماسح الكود العام، فهو يتعلق تحديدا بوجود `sanitize()` *الخاصة بهذه المكتبة* في المسار قبل عرض قيمة تبدو خطرة.

```js
// eslint.config.js
import unicodeShield from 'unicode-shield/eslint-plugin'

export default [
  {
    plugins: {'unicode-shield': unicodeShield},
    rules: {'unicode-shield/require-sanitized-text': 'warn'},
  },
]
```

```jsx
<Text>{user.displayName}</Text>            // ترصد

<Text>{sanitize(user.displayName)}</Text>  // سليمة

const safeName = sanitize(user.displayName)
<Text>{safeName}</Text>                    // سليمة، تم تتبعها حتى استدعاء sanitize()
```

### `require-sanitized-text`

ترصد هذه القاعدة تعبير JSX عندما يكون معرفا مباشرا أو وصولا إلى خاصية باسم معروف على نحو ثابت، سواء بنقطة أو بقوس (`{username}`, `{user.bio}`, `{user["bio"]}`)، ويطابق اسمه قائمة من أسماء شبيهة بالهوية معدة مسبقا، ما لم يكن ملفوفا في `sanitize(...)` مباشرة، أو تم تتبعه حتى تصريح متغير واحد بالضبط نشأ من استدعاء `sanitize(...)`.

| الخيار | القيمة الافتراضية | |
| --- | --- | --- |
| `riskyNames` | `['username', 'handle', 'displayname', 'nickname', 'bio']` | مطابقة substring دون حساسية لحالة الأحرف، مع اسم المعرف أو الخاصية. تستبدل القائمة الافتراضية ولا تضيف إليها. |
| `sanitizerNames` | `['sanitize']` | أسماء دوال، سواء مباشرة أو كخاصية (مثل `unicodeShield.sanitize`)، تعرف بوصفها دوالا تطهر وسيطها. |

**ما لا تقوم به هذه القاعدة عمدا**، وهو نطاق الإصدار الأول لا نسيان:

- تفحص فقط *محتوى* JSX (`<Text>{x}</Text>`)، لا الخصائص (`alt={x}`, `title={x}`). خصائص العرض هذه فجوة معروفة.
- تفهم فقط معرفا مباشرا أو وصولا واحدا إلى خاصية (مفتاح محسوب مثل `user[someVariable]` لا يمكن حله على نحو ثابت فيتجاهل)، وتتبع المتغير حتى تصريح واحد بالضبط. وهي heuristic قائمة على الاسم، لا تحليل data-flow حقيقي: `{formatHandle(handle)}` لا تفحص، وإعادة التعيين (`let x = a; x = b`) لا تتبع.
- تطابق الأسماء بـsubstring دون حساسية لحالة الأحرف، فاسم مثل `usernamePattern` قد يرصد خطأ. يمكن تغيير اسم المتغير، أو تعديل `riskyNames`، أو تعطيل السطر.
- تعترف فقط باستدعاء `sanitize()` (أو اسم أضيف إلى `sanitizerNames`) دليلا على الأمان، لا بحارس `isSafe()`.

تتطلب `eslint >= 9`، وهي معلنة كـpeer dependency اختيارية: الـAPI الأساسي `scan`/`sanitize`/`isSafe` لا يعتمد على ESLint إطلاقا، هذا المسار الفرعي فقط هو الذي يعتمد عليه.

## ما لا تكون هذه المكتبة عليه

هذه ليست ماسح كود مصدري، ولا عامل تصفية ألفاظ، ولا كاشف تشابه بصري (homoglyph). فهي محصورة في فئات bidi والمحارف المخفية المذكورة أعلاه. اكتشاف التشابه البصري أمر حقيقي ومفيد ومشكلة كبيرة بما يكفي (آلاف التعيينات المعرفة في يونيكود) لتستحق إصدارا منفصلا بدلا من إضافة غير مكتملة هنا.
