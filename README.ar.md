# unicode-shield

[![CI](https://github.com/a-y-ibrahim/unicode-shield/actions/workflows/ci.yml/badge.svg)](https://github.com/a-y-ibrahim/unicode-shield/actions/workflows/ci.yml)

[![English](https://img.shields.io/badge/English-Click-blue)](README.md)

مكتبة لكشف محارف يونيكود الخطرة في النصوص التي يكتبها المستخدمون وإزالتها، دون أن تفسد نصا عربيا أو عبريا حقيقيا أو غيرهما من لغات الكتابة من اليمين إلى اليسار، ودون أن تؤثر في رموز الإيموجي.

```bash
npm install unicode-shield
```

## لماذا هذه المكتبة

توجد محارف في يونيكود تظهر على الشاشة وكأنها غير موجودة، أو تجعل النص يظهر بشكل مختلف عن حقيقته. اسم مستخدم، أو رسالة في محادثة، أو اسم ملف كتبه شخص خبيث، قد يخفي محارف من هذا النوع:

- **تعكس اتجاه عرض النص**، باستخدام محارف bidi embedding/override، وهي الآلية نفسها التي استند إليها الكشف الأمني المعروف باسم "Trojan Source" عام 2021 ([CVE-2021-42574](https://nvd.nist.gov/vuln/detail/CVE-2021-42574)). وهذه فئة الثغرة نفسها التي أصلحت في [`bluesky-social/social-app#11066`](https://github.com/bluesky-social/social-app/pull/11066): معرف مستخدم مثل `admin` يمكن أن تضاف إليه محارف مخفية، فيظهر عند نسخه ولصقه كنص مختلف تماما.
- **تحشو أو تكرر نصوص الهوية**، باستخدام محارف عديمة العرض، بحيث يظهر اسما مستخدمين متطابقين بصريا وكأنهما مختلفان، أو تتجاوز كلمة محظورة عامل تصفية.
- **تهرب تعليمات مخفية**، باستخدام كتلة Unicode Tags المهجورة (U+E0000-U+E007F) أو Variation Selectors Supplement (U+E0100-U+E01EF)، وكلتاهما استخدمتا منذ عام 2024 كوسيلة لحقن أوامر خفية. نص مشفر في هذه النقاط يظهر فارغا في أي خط عادي، لكن بعض نماذج الذكاء الاصطناعي ما زالت تقرأه وتتصرف بناء عليه.

## لماذا لا نستخدم أداة موجودة بالفعل

توجد ماسحات للكود المصدري لهذه الفئة من المحارف، وهي مبنية جيدا، وعلى رأسها [`anti-trojan-source`](https://github.com/lirantal/anti-trojan-source). لكنها تحل مشكلة مختلفة: اكتشاف هذه المحارف *داخل الكود نفسه*. أما `unicode-shield` فمخصصة للجانب الآخر: اكتشافها *في البيانات التي يكتبها المستخدمون*، لحظة تخزينها أو عرضها.

قبل بناء هذه المكتبة بحثت عن مكتبة تعمل وقت التشغيل وتقوم بالمهمة نفسها، عبر npm وPyPI وcrates.io وpkg.go.dev، وكذلك أدوات الأمان مفتوحة المصدر التي تنشرها كبرى شركات الذكاء الاصطناعي. وما وجدته لم يكن كافيا. بعض الأدوات ضيقة النطاق، تقتصر على إزالة المحارف عديمة العرض أو على اكتشاف التشابه البصري فقط. وبعضها ماسح كود مصدري أو CI يعمل على الطبقة الخاطئة. أما المشاريع الحديثة جدا فتكتشف المحارف الصحيحة، لكنها تحذف المحارف الشرعية أيضا، فتفسد نص RTL والإيموجي اللذين تحرص هذه المكتبة على عدم المساس بهما.

لم أجد أداة تجمع بين اكتشاف bidi، واكتشاف كتلة Unicode Tags، والحفاظ المتعمد على علامات الاتجاه الشرعية وروابط النص. وهذا الجمع، بما في ذلك الدقة في الجانب الآمن، هو ما بنيت هذه المكتبة من أجله.

## الجانب الآخر من المشكلة

من السهل أن تخطئ أداة كهذه في الاتجاه المعاكس، فتحذف كل شيء غير مألوف وتفسد نصا حقيقيا من غير أن تلاحظ. وللعديد من المحارف المذكورة أعلاه استخدامات شرعية تماما:

- `‎` (U+200E، LEFT-TO-RIGHT MARK) و`‏` (U+200F، RIGHT-TO-LEFT MARK) تلميحا اتجاه بمحرف واحد، يعتمد عليهما فعلا النص العربي والعبري وغيرهما من لغات RTL الصحيحة.
- `‍` (U+200D، ZERO WIDTH JOINER) هو الطريقة التي يبنى بها إيموجي مركب مثل 👨‍👩‍👧‍👦، وهو ضروري لتشكيل النص الصحيح في عدة لغات.
- `‌` (U+200C، ZERO WIDTH NON-JOINER) ضروري لتكوين الكلمة الصحيح في الفارسية وعدة لغات هندية.

تبلغ `unicode-shield` عن هذه المحارف أيضا، لأن الرؤية الكاملة لمحتوى النص مفيدة، لكنها لا تحذفها أبدا افتراضيا. ولا تحذف `sanitize()` سوى الفئات التي ليس لها استخدام شرعي في نص قصير كتبه مستخدم.

## طريقة الاستخدام

```ts
import {scan, sanitize, isSafe} from 'unicode-shield'

isSafe('admin')                    // true
isSafe('admin\u{202E}nimda')       // false، يحتوي على bidi override

sanitize('admin\u{202E}nimda')     // 'adminnimda'

scan('price: 100\u{200E}\u{200F} ريال\u{061C}')
// {
//   safe: true,   // علامات bidi معلوماتية فقط، لا خطر حقيقي
//   threats: [
//     { category: 'bidi-mark', severity: 'informational', codePoint: 0x200e, ... },
//     { category: 'bidi-mark', severity: 'informational', codePoint: 0x200f, ... },
//     { category: 'bidi-mark', severity: 'informational', codePoint: 0x61c, ... },
//   ]
// }
```

### `scan(input: string): ScanResult`

تعيد كل تهديد موجود في `input`، الخطير منه والمعلوماتي. ولا تكون `safe` بقيمة `true` إلا إذا لم يوجد أي تهديد خطر.

### `sanitize(input: string, options?: SanitizeOptions): string`

تعيد `input` بعد إزالة المحارف الخطرة منه. تحذف افتراضيا فئات `bidi-embedding` و`bidi-isolate` و`invisible` و`tag` و`variation-selector` و`combining-marks`. أما الفئات المعلوماتية (`bidi-mark`, `joiner`) فلا تمس إلا إذا طلب ذلك صراحة:

```ts
sanitize(input, {categories: ['bidi-mark']})   // تحذف أيضا LRM/RLM/ALM
sanitize(input, {replacement: '�'})       // تستبدل بدلا من أن تحذف
```

### `isSafe(input: string): boolean`

اختصار لـ `scan(input).safe`.

## فئات التهديد

| الفئة | الخطورة | أمثلة | تحذف افتراضيا؟ |
| ----------------- | --------------- | -------------------------------------- | -------------------- |
| `bidi-embedding`  | خطرة       | LRE, RLE, LRO, RLO, PDF                | نعم |
| `bidi-isolate`    | خطرة       | LRI, RLI, FSI, PDI                     | نعم |
| `invisible`       | خطرة       | zero-width space, word joiner, stray BOM | نعم |
| `tag`             | خطرة       | U+E0000-U+E007F (كتلة Tags المهجورة) | نعم |
| `variation-selector` | خطرة    | U+E0100-U+E01EF (Variation Selectors Supplement) | نعم |
| `combining-marks` | خطرة       | تكديس علامات تشكيل يتجاوز حدا آمنا ("Zalgo text") | نعم |
| `bidi-mark`       | معلوماتية   | LRM, RLM, ALM                          | لا |
| `joiner`          | معلوماتية   | ZWJ, ZWNJ                              | لا |

ملاحظة حول `variation-selector`: كتلة الـSupplement فقط هي المشمولة. أما كتلة Variation Selectors الأساسية (U+FE00-U+FE0F، VS15/VS16) فلا تعد خطرة أبدا، لأنها الطريقة التي يختار بها النص العادي عرضا نصيا أو عرض إيموجي لآلاف المحارف الشائعة، وهي شائعة جدا في نصوص المستخدمين الحقيقية.

ملاحظة حول `combining-marks`: كل محرف في يونيكود من فئة General_Category المسماة Mn (علامة تشكيل غير متباعدة) يعد علامة تشكيل هنا، والقائمة مولدة من بيانات يونيكود نفسها لا من قائمة انتقيت يدويا. الخطر ليس في المحرف نفسه، فنصوص حقيقية في عشرات أنظمة الكتابة تعتمد على علامات التشكيل، بل في تكديس أكثر من 6 علامات فوق حرف أساس واحد، وهذه هي الحيلة التي تقوم عليها إساءة استخدام "نص Zalgo" (تشويه بصري يستخدم للمضايقة أو إفساد اسم مستخدم أو رسالة). النصوص الحقيقية المشكولة بكثافة، كالعربية المشكولة بالكامل أو العبرية بنقطها وعلامات ترتيلها، تبقى بعيدة تماما عن هذا الحد. و`sanitize()` لا تحذف الحد الزائد كله، بل تبقي 6 علامات على الحرف الأساس وتحذف ما بعدها، فيحتفظ الحرف بقدر معقول وواقعي من الزخرفة.

## إضافة ESLint

لا تفيد `scan` و`sanitize` إلا إذا استدعاهما أحد فعلا. وتوجد طريقة أخرى تصل بها هذه الفئة من الأخطاء إلى الإنتاج: مسار بيانات لا يمر بأي منهما إطلاقا. وهنا يأتي دور `unicode-shield/eslint-plugin`: يكتشف هذه الحالة وقت المراجعة بدلا من الإنتاج، بالتحقق من وصول نص شبيه بالهوية، مثل اسم مستخدم أو handle أو اسم عرض أو bio، إلى JSX من غير تطهير، سواء كان محتوى معروضا أو قيمة خاصية تعرض نصا مثل `alt` أو `title` أو `placeholder` أو `aria-label` أو `value`.

هذا فحص لبنية الكود، من النوع الموصوف أعلاه في "لماذا لا نستخدم أداة موجودة بالفعل" بوصفه ماسح كود مصدري أو CI، وليس الفحص وقت التشغيل الذي تقوم به `scan()`. وعلى عكس ماسح الكود العام، فهو يتعلق تحديدا بأمر واحد: هل `sanitize()` الخاصة بهذه المكتبة موجودة في المسار قبل أن تظهر قيمة تبدو خطرة؟

```js
// eslint.config.js
import unicodeShield from 'unicode-shield/eslint-plugin'

export default [
  {
    plugins: {'unicode-shield': unicodeShield},
    rules: {'unicode-shield/require-sanitized-text': 'warn'},
  },
]
```

```jsx
<Text>{user.displayName}</Text>            // ترصد، محتوى غير مطهر
<img alt={user.bio} />                     // ترصد، خاصية غير مطهرة

<Text>{sanitize(user.displayName)}</Text>  // سليمة

const safeName = sanitize(user.displayName)
<Text>{safeName}</Text>                    // سليمة، تم تتبعها حتى استدعاء sanitize()
```

### `require-sanitized-text`

ترصد هذه القاعدة محتوى JSX أو قيمة خاصية نصية فيه، عندما يكون معرفا مباشرا، أو وصولا إلى خاصية باسم ثابت معروف مسبقا، سواء بنقطة أو بقوس، مثل `{username}` أو `{user.bio}` أو `{user["bio"]}`. والشرط أن يطابق اسمه قائمة أسماء شبيهة بالهوية معدة مسبقا.

ولا ترصد القاعدة هذه الحالة في موضعين: إذا كان ملفوفا في `sanitize(...)` مباشرة، أو إذا تم تتبعه حتى تصريح متغير واحد بالضبط نشأ من استدعاء `sanitize(...)`.

| الخيار | القيمة الافتراضية | |
| --- | --- | --- |
| `riskyNames` | `['username', 'handle', 'displayname', 'nickname', 'bio']` | مطابقة substring دون حساسية لحالة الأحرف، مع اسم المعرف أو الخاصية. تستبدل القائمة الافتراضية ولا تضيف إليها. |
| `riskyAttributes` | `['alt', 'title', 'placeholder', 'aria-label', 'value']` | مطابقة تامة دون حساسية لحالة الأحرف مع اسم خاصية JSX. هذه الخصائص فقط هي المفحوصة، أي خاصية أخرى (`className`, `href`, `onClick`, ...) ليست منفذا نصيا وتترك دون فحص. تستبدل القائمة الافتراضية ولا تضيف إليها. |
| `sanitizerNames` | `['sanitize']` | أسماء دوال، سواء مباشرة أو كخاصية (مثل `unicodeShield.sanitize`)، تعرف بوصفها دوالا تطهر وسيطها. |
| `autoImport` | `{name: 'sanitize', source: 'unicode-shield'}` | ما يلف --fix حوله المخالفة، ومصدر استيراده، تضاف أو تدمج مع استيراد قائم من المصدر نفسه حسب الحاجة. عطلها بـfalse للاكتفاء بالرصد دون عرض إصلاح. |

تدعم القاعدة --fix: تلف القيمة المرصودة داخل استدعاء، فتتحول `user.bio` إلى `sanitize(user.bio)`، وتضيف الاستيراد اللازم أو تدمجه مع استيراد قائم من المصدر نفسه. بهذا يغلق تشغيل `eslint --fix`، أو الإصلاح التلقائي عند الحفظ في المحرر، معظم المخالفات دون تعديل يدوي لكل موضع على حدة.

وجه `autoImport` إلى دالتك الخاصة، مثل `{name: 'cleanText', source: '~/utils/text'}`، إن كان مشروعك يطهر النص بطريقة أخرى غير استدعاء `sanitize()` من هذه الحزمة مباشرة.

**ما لا تقوم به هذه القاعدة عمدا**، وهو نطاق الإصدار الأول لا نسيان:

- تفحص محتوى JSX وقائمة محددة مسبقا من الخصائص النصية، لا أي خاصية عشوائية. خاصية ليست ضمن `riskyAttributes`، خاصية `label` في مكون مخصص مثلا، لا تفحص إطلاقا، أضفها إلى `riskyAttributes` إن احتاجت التغطية نفسها.
- تفهم فقط معرفا مباشرا أو وصولا واحدا إلى خاصية. مفتاح محسوب مثل `user[someVariable]` لا يمكن تحديد قيمته مسبقا فيتجاهل. وتتبع المتغير حتى تصريح واحد بالضبط. فهي heuristic قائمة على الاسم، لا تحليل data-flow حقيقي: `{formatHandle(handle)}` لا تفحص، وإعادة التعيين (`let x = a; x = b`) لا تتبع.
- تطابق الأسماء بـsubstring دون حساسية لحالة الأحرف، فاسم مثل `usernamePattern` قد يرصد خطأ. يمكن تغيير اسم المتغير، أو تعديل `riskyNames`، أو تعطيل السطر.
- تعترف فقط باستدعاء `sanitize()` (أو اسم أضيف إلى `sanitizerNames`) دليلا على الأمان، لا بحارس `isSafe()`.
- الإصلاح يلف القيمة المرصودة ويدير استيرادا واحدا فقط، ولا يعيد تنسيق الملف. شغل أداة التنسيق الخاصة بمشروعك بعده إن لم يطابق أسلوب الاستيراد المضاف بقية الملف.
- الإصلاح يضيف استيراد ES فقط، فلا يفيد في ملف CommonJS. ولا شيء غيره يحمي من إصلاح غير آمن يكسر الملف: `--fix` في ESLint يكتب الناتج كما هو، دون أي تحقق من أنه لا يزال قابلا للتحليل. لهذا تتحقق هذه القاعدة بنفسها من حالتين قبل عرض أي إصلاح.

  الأولى: أن يكون `autoImport.name` معرفا صحيحا نحويا وغير محجوز. خطأ إملائي مثل `sanitize-text`، أو اسم مثل `class`، يجعل القاعدة تكتفي بالرصد دون إصلاح.

  الثانية: ألا يكون هذا الاسم مرتبطا مسبقا بشيء آخر في الملف، استيراد مختلف، أو متغير، أو دالة محلية بالاسم نفسه. في هذه الحالة أيضا تكتفي القاعدة بالرصد، بدلا من التصادم مع الاسم أو استدعاء شيء غير مقصود بصمت.
- يتجاهل بشكل صحيح `import type {...}` في TypeScript، أو الصيغة الجزئية `import {type x}`، عند تحديد ما إذا كان `autoImport.name` مستوردا بالفعل أو هدفا صالحا للدمج. فهذه الصيغ لا تربط أي قيمة وقت التشغيل، والاعتماد عليها ينتج إصلاحا لا يعمل بصمت، فيضاف بدلا منه استيراد قيمة حقيقي.

تتطلب `eslint >= 9`، وهي معلنة كـpeer dependency اختيارية: الـAPI الأساسي `scan`/`sanitize`/`isSafe` لا يعتمد على ESLint إطلاقا، هذا المسار الفرعي فقط هو الذي يعتمد عليه.

## التشابه البصري (كشف المحارف المتشابهة الشكل)

تغطي `scan` و`sanitize` المحارف التي تخفي النص أو تعكس اتجاهه. لكنهما لا تغطيان النصف الآخر من تزييف يونيكود: محارف مصممة لتشبه محرفا آخر شكلا. وهذا أساس الخداع في النطاقات وأسماء المستخدمين المتشابهة شكلا، مثل كلمة apple مكتوبة بحرف a سيريلي بدل اللاتيني. هذا مسار فرعي منفصل واختياري، `unicode-shield/confusables`، مبني على بيانات الأمان الرسمية من يونيكود نفسه (معيار UTS رقم 39)، لا على قائمة انتقيت يدويا:

```ts
import {getSkeleton, areConfusable, detectMixedScript} from 'unicode-shield/confusables'

areConfusable('apple', 'аpple')   // true: الحرف a الثاني سيريلي
areConfusable('apple', 'orange')       // false

detectMixedScript('аpple')
// { mixed: true, scripts: ['Cyrillic', 'Latin'], suspicious: [{ char: 'а', codePoint: 0x430, index: 0, script: 'Cyrillic' }] }
```

### `getSkeleton(input: string): string`

الصورة القياسية لنص ما حسب معيار UTS 39: تطبيع يونيكود، ثم استبدال كل محرف له نظير موثق في بيانات التشابه بذلك النظير، ثم تطبيع مرة أخرى. يتشابه نصان بصريا بالضبط عندما تتطابق صورتاهما القياسيتان.

### `areConfusable(a: string, b: string): boolean`

اختصار لـ `getSkeleton(a) === getSkeleton(b)`. الاستخدام العملي الحقيقي: التحقق مما إذا كان اسم مستخدم أو اسم عرض جديد يصعب تمييزه بصريا عن اسم موجود فعلا، وهذه بالضبط فئة الثغرة التي بدأت منها قصة هذه المكتبة.

### `detectMixedScript(input: string): MixedScriptResult`

ترصد هذه الدالة نصا يخلط بين نظامي كتابة أو أكثر، بخلاف الفئتين Common وInherited (الأرقام وعلامات الترقيم والمحارف التوليفية مشتركة بين كل أنظمة الكتابة، فلا تعد خلطا بحد ذاتها). وهذا هو النمط الذي يقوم عليه تزييف النطاقات المتشابهة شكلا. وتوضح النتيجة نظام الكتابة الأغلب، والمحارف بالتحديد التي لا تنتمي إليه. هذه heuristic عملية، وليست تطبيقا كاملا لخوارزمية مستويات التقييد في معيار UTS 39.

**مثال حقيقي يوضح لماذا معيار الحكم ليس "يبدو غريبا".** بيانات يونيكود نفسها تدرج الألف العربية (ا) ضمن المحارف المتشابهة بصريا مع اللاتينية `l`، فكلاهما خط رأسي بسيط. وستؤكد `getSkeleton` و`areConfusable` ذلك بشكل صحيح. وبالمثل، ستعطي `detectMixedScript('مرحبا')` بشكل صحيح أيضا النتيجة `mixed: false`، لأن كل محرف في هذه الكلمة عربي. التشابه البصري والانتماء لنظام كتابة معين سؤالان مستقلان تماما، والخلط بينهما هو بالضبط ما يجعل أداة كشف التزييف تخطئ في تصنيف نص RTL عادي كمشبوه.

**لماذا مسار فرعي منفصل.** بيانات التشابه وأنظمة الكتابة في يونيكود تغطي آلاف التعيينات، والبيانات المولدة التي تشحنها هذه الميزة نحو 190 كيلوبايت. إبقاؤها خارج الاستيراد الأساسي لـ`unicode-shield` يعني أن مستخدمي `scan` و`sanitize` الذين لا يحتاجون كشف التشابه البصري لا يدفعون أي تكلفة إضافية مقابله: حجم `dist/index.js` غير متأثر بحجم هذه الميزة إطلاقا.

## سطر الأوامر (CLI)

كل ما سبق واجهة برمجية تستخدم داخل الكود. لكن تثبيت `unicode-shield` يوفر أيضا أداة سطر أوامر، لفحص الملفات والمجلدات دون كتابة أي كود، مثلا داخل خطوة CI تفحص محتوى من إنتاج المستخدمين أو ملفات ترجمة.

```bash
npx unicode-shield scan ./content
npx unicode-shield scan file.txt --json
npx unicode-shield sanitize file.txt > clean.txt
npx unicode-shield sanitize ./content --write
npx unicode-shield compare "apple" "аpple"

# - تعني stdin، نفس الاصطلاح المعروف في grep وjq، لبناء أنابيب Unix حقيقية
cat file.txt | npx unicode-shield scan -
some-tool | npx unicode-shield sanitize - | another-tool
```

### `unicode-shield scan <path>`

تفحص ملفا واحدا، أو كل ملف داخل مجلد بشكل متكرر (تتجاوز `node_modules` و`.git` وعددا من مجلدات البناء والتبعيات الأخرى، وقائمة محددة من امتدادات الملفات الثنائية)، أو stdin عندما يكون المسار `-`، وتبلغ عن كل تهديد ترصده `scan()`. تخرج بالرمز `1` إذا وجد أي تهديد خطر، و`0` إذا كان كل شيء سليما. خيار `--json` يطبع مخرجات منظمة بدل الشكل المقروء الافتراضي.

### `unicode-shield sanitize <path>`

تشغل `sanitize()` على ملف أو مجلد أو stdin عندما يكون المسار `-`. بلا `--write`، تطبع المحتوى بعد التطهير إلى stdout، فيمكن تمريره عبر أنبوب: `unicode-shield sanitize in.txt > out.txt`. أما `--write` فتعدل الملف أو الملفات في مكانها مباشرة: إلزامي عند فحص مجلد كامل (إذ لا يوجد مجرى stdout واحد يصلح لطباعة عدة ملفات فيه بشكل مفيد)، ومرفوض مع stdin (إذ لا يوجد أصلا ملف تكتب إليه). وخياري `--replacement <str>` و`--categories <a,b,c>` ينقلان مباشرة إلى خيارات `sanitize()` نفسها، واسم فئة غير معروف يرفض مع عرض القائمة الصحيحة بدل تجاهله بصمت.

### `unicode-shield compare <a> <b>`

تشغل `areConfusable()` على نصين مباشرة، للتحقق مما إذا كان اسم مستخدم أو اسم عرض جديد يصعب تمييزه بصريا عن اسم موجود فعلا. تخرج بالرمز `1` إذا كانا متشابهين، و`0` إن لم يكونا. خيار `--json` يضيف أيضا صورتيهما القياسيتين (skeleton).

رموز الخروج في الأوامر الثلاثة كلها: `0` سليم، `1` وجد تهديد أو زوج متشابه، `2` خطأ في الاستخدام أو أثناء التشغيل (وسيطات خاطئة، مسار غير موجود).

## ما لا تكون هذه المكتبة عليه

هذه ليست ماسح كود مصدري ولا عامل تصفية ألفاظ. كشف التشابه البصري وخلط أنظمة الكتابة أعلاه يغطيان فئة محددة وواضحة من التزييف البصري، وليسا مصنفا عاما لـ"هل هذا النص مريب"، ولا يحاولان أن يكونا كذلك عمدا.
